## Тема сессии
Деплой Lead Machine на VPS Beget Latvia (91.193.25.237)

## Сделано

### Доступ к серверу
- Прямой SSH с этой машины не проходил (нет ключа в authorized_keys, нет `expect`)
- Решение: PuTTY `plink`/`pscp` с паролем (`-pw`), уже установлен в системе — рабочий аналог expect для Windows
- Подключились под root после двух попыток пароля

### Подготовка сервера
- Диск был забит на 99% (153 МБ свободно) — почистили мёртвый кэш VS Code Tunnel (`/root/.vscode/cli/servers`, 2.9 ГБ, туннель даже не был запущен), apt-кэш, журналы. Освободили 3.3 ГБ
- Установили Docker (get.docker.com) — его не было

### Деплой кода
- Repo `lead-machine` приватный — git clone на сервере не прошёл (нет токена там). Перенесли тарболлом через pscp вместо этого
- Перенесли уже авторизованную Telethon-сессию (`data/sender_session.session`) — повторная авторизация не понадобилась
- Поправили `docker-compose.prod.yml`: app использует `network_mode: host`, поэтому db/redis нужны `ports: 127.0.0.1:5432/6379` — без этого app не достучался бы до localhost:5432/6379
- Добавили `DB_PASSWORD` в `.env` для подстановки `${DB_PASSWORD}` в compose
- Поймали баг: `Dockerfile` делает `COPY . .`, значит `.env` копируется в образ при сборке — любая правка .env требует пересборки
- Поймали баг в коде: aiogram 3.7.0 больше не принимает `parse_mode=` в `Bot()` — нужен `default=DefaultBotProperties(...)`. Исправили `app/bot/control.py`, закоммитили и запушили в github.com/slavinbiz/lead-machine
- Добавили `extra="ignore"` в `Settings.model_config` (app/config.py), чтобы лишний DB_PASSWORD не валил pydantic-валидацию

### Результат
- FastAPI + dispatch_loop запущены и работают
- Alembic-миграции прошли (initial schema)
- DB (postgres:16) и Redis — контейнеры здоровы
- **Не работает:** Control Bot — `TELEGRAM_BOT_TOKEN` в `.env` оказался плейсхолдером, не настоящим токеном. Падает `TelegramUnauthorizedError`

## Решения

- На этой Windows-машине для SSH с паролем — PuTTY plink/pscp, не expect (его нет)
- Приватные репо на сервер — тарболлом через pscp, не git clone (нет токена на сервере)
- Все детали и команды для повтора/дебага — knowledge/lead-machine-deploy.md

## Сессия 2
Тема: личный VPN для VS Code/Claude Code + Telegram + Chrome (без Happ)

### Сделано
- Вячеслав решил отказаться от Happ и гонять VS Code/Claude Code, Telegram и часть сайтов в Chrome через свой Hysteria2-сервер; остальной браузинг — Яндекс без VPN
- Отключили Happ для проверки — выяснилось: прямое подключение (без VPN) у Anthropic API и GitHub работает нормально, но **claude.com (страница логина) блокирует российские IP** — без VPN с не-российским адресом перелогиниться в Claude Code нельзя
- v2rayN (Windows GUI) оказался ненадёжным: окно открывается, но соединение не активируется и процесс тихо закрывается (видно в `guiLogs` — старт → закрытие, без единой строки про коннект). Обошли через консоль: подняли `sing-box.exe` (он уже внутри v2rayN, `bin/sing_box/`) напрямую со своим JSON-конфигом
- Старая личная ссылка Hysteria2 (`53dbf1a75aa7d83992b63567cdf866c2@...`) — мёртвая, сервис `hysteria-server.service` на сервере отключён (decommissioned), порт UDP/443 теперь полностью у **s-ui** (проект «VPN на продажу»)
- Тестовый клиент `test-july` (пароль `JITQrtrP3D`) тоже не подключался — `authentication failed, status code: 404`. Зашли по SSH (PuTTY plink, пароль root дал Вячеслав в чате — не сохранён) и нашли причину в sqlite `/usr/local/s-ui/db/s-ui.db`: у клиента `test-july` стоял `enable=0`. **В интерфейсе s-ui для этого поля нет переключателя** — Вячеслав подтвердил, что в панели клиент показан как активный. Включили вручную через `UPDATE clients SET enable=1 WHERE name='test-july';` + `systemctl restart s-ui`
- После этого туннель заработал: `curl -x socks5h://127.0.0.1:10810 https://api.ipify.org` → `64.188.57.249`, `CF-RAY` показывает Frankfurt
- Локальный прокси сейчас: `127.0.0.1:10810` (SOCKS5+HTTP), процесс `sing-box.exe`, конфиг `C:\v2rayN\hy2-client\config.json`, запущен как фоновый процесс в текущей сессии (не автозапуск, переживёт только до перезагрузки/закрытия)

### Решения
- v2rayN GUI не использовать для активации — ненадёжен на этой машине. Источник истины — `sing-box.exe` с ручным конфигом
- ⚠️ **Важная находка для проекта «VPN на продажу»**: у новых клиентов в s-ui поле `enable` в базе может быть 0 без видимой возможности включить через UI — это блокер для реальных платных клиентов, нужно разобраться до первой продажи (баг s-ui или мы что-то не так создаём клиента)
- SSH на TimeWeb-сервер — тоже через PuTTY plink с паролем (как и для Beget сегодня), ключа нет
