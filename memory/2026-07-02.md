## Сессия 2 (с компьютера) — попытка добавить obfs уронила VPN всем

Параллельная сессия Jarvis (через Telegram-бота на Beget) полезла выполнять TODO выше — зашла на TimeWeb по SSH и стала руками через `sqlite3` менять таблицу `inbounds` в `/usr/local/s-ui/db/s-ui.db`, добавляя `obfs: salamander` в колонку `options`. Сессия оборвалась по сети (`API Error: ECONNRESET`) прямо на рестарте `s-ui` — VPN лёг полностью: и мой личный туннель (test-july), и все клиенты бота (`vpnbot_888224075` и др.).

**Причина поломки (двойная):**
1. `sqlite3` CLI пишет JSON-строки в blob-колонки (`options`, `out_json`) как TEXT — Go-код s-ui падает на `sql: Scan error ... unsupported Scan, storing driver.Value type string into type *json.RawMessage`, ядро Hysteria2 не стартует (уходит в вечный цикл `starting core` каждые 5 сек, порт UDP/443 не слушает). Это уже вторая поломка s-ui за 2 дня по этой причине (утром 1 июля — та же ошибка из-за мусорной записи `test-trigger`)
2. obfs включили только на сервере — ни мой Windows-клиент (`C:\v2rayN\hy2-client\config.json`), ни клиенты бота не знают про obfs, поэтому даже после починки ядра никто не мог подключиться

**Починил:**
- Бэкап базы (`s-ui.db.bak-*-colleague-fix`)
- `UPDATE inbounds SET out_json = CAST(out_json AS BLOB), options = CAST(options AS BLOB) WHERE id=1` — почему это чинит: SQLite сам меняет storage class на BLOB, Go-драйвер начинает отдавать `[]byte` вместо `string`
- Убрал obfs из `options` обратно (несогласованное включение ломало всех платных клиентов) — вернул чистый `{"listen":"::","listen_port":443}`
- Перезапустил sing-box-клиент на Windows, проверил curl через SOCKS5 — 200 OK
- Убедился что реальный клиент бота (`vpnbot_888224075`) снова коннектится

## Решения
- **Задачу "добавить obfs salamander" делать НЕ через сырой sqlite3**, а через API/UI s-ui, или если только через SQL — оборачивать JSON в `CAST(... AS BLOB)` сразу при INSERT/UPDATE
- Включать obfs на сервере — только вместе с одновременным обновлением ВСЕХ клиентов (мой личный + шаблон для клиентов vpn-бота), иначе рвёт всем текущие подключения. Не делать это в одиночной сессии без плана синхронного роллаута
- Параллельные Jarvis-сессии (с компьютера и через Telegram-бота) могут одновременно лезть в один и тот же сервер/базу — при работе с VPN-инфраструктурой сверяться со свежими дневниковыми записями перед правками, не полагаться что ты единственный, кто сейчас там копается

## Сессия 5 — порт сменили на 8443, упёрлись в облачный файрвол TimeWeb

Session 4 (параллельная) заодно перенесла Hysteria2 с порта 443 на **8443** (server + client конфиг синхронно, всё чисто). После перезапуска sing-box.exe с новым конфигом соединения всё равно не было — `ss -ulnp` на сервере показал, что ядро слушает 8443 нормально, OS-файрвол (ufw) выключен, значит дело в **облачном файрволе TimeWeb** (отдельная штука от ufw/iptables, по умолчанию открыт только 22/TCP + 443/TCP+UDP, см. knowledge/infrastructure.md). Вячеслав вручную открыл UDP/8443 в панели timeweb.cloud → сразу заработало, curl подтвердил 200.

**Итог по VPN на конец дня 2 июля (обновлено):** Hysteria2 (UDP) работает на 443 с obfs на Windows, но **на Tele2 мобильном не работает в принципе** — подтверждено синхронным `tcpdump` на сервере: пакеты с телефона не доходят вообще ни на одном порту/протоколе. Tele2 глушит исходящий UDP с мобильного данных подчистую (не DPI-фингерпринтинг, а блэкхол), obfs тут бессилен.

**Решение:** для мобильного клиента нужен TCP-протокол, не UDP. Остановил старый `x-ui`/xray-Reality на TCP/443 (не работал, DPI-заблокирован ещё раньше) — освободил порт. Добавил в s-ui **Trojan-инбаунд** на TCP/443 (TLS, тот же самоподписанный сертификат `/etc/hysteria/server.crt`, что и у Hysteria2), привязал клиента `test-july` (`clients.inbounds = [1,2]`). Sing-box поднял оба инбаунда одновременно: `hysteria2[hysteria2-443] udp` + `trojan[trojan-443] tcp` на одном порту 443, разных протоколах. TCP/443 снаружи доступен (проверено curl-рукопожатием). Дал Вячеславу trojan:// QR для второго профиля в v2rayNG — ждём проверки на Tele2.

**Бэкапы БД за сегодня:** `s-ui.db.bak-202607011505` (утро, test-trigger), `*-colleague-fix` (после первого obfs-инцидента), `*-trojan` (перед добавлением trojan-инбаунда).

## Сессия 6 — финальный диагноз: IP сервера в блоке у Tele2 целиком

Добавил Trojan-инбаунд (TCP/443, TLS) специально для мобильного — на Wi-Fi заработал сразу. На Tele2 всё равно таймаут. Проверили прямым тестом: открыть `http://64.188.57.249:2095` в браузере телефона на Tele2 (без VPN, без протокола VPN вообще) — **страница не грузится**. Значит проблема не в Hysteria2/obfs/Trojan/портах — Tele2 блокирует сам IP `64.188.57.249` целиком на уровне сети. Дальше менять протокол/порт бессмысленно.

**Единственный надёжный выход:** спрятать IP за Cloudflare — завести домен, прокси-режим (оранжевое облако), привязать SNI Trojan-инбаунда к этому домену, клиент подключается по domain:443, а не по IP. Cloudflare блокировать не будут (заденет полинтернета). Нужен домен — у Вячеслава сейчас нет свободного под это, решение отложено.
