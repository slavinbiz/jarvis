# INSTALL.md — инструкция для Claude Code

> **Это служебный файл.** Его читает Claude Code, когда ученик курса «Архитектор нейросотрудников» просит установить архитектуру Агента. После завершения установки этот файл удаляется.

---

## Кто ты сейчас

Ты — Claude Code, открытый в папке проекта ученика. Ученик только что попросил установить архитектуру Агента из шаблона `Ntmib/jarvis-architect`.

К моменту чтения этого файла шаблон уже скачан (через `npx degit Ntmib/jarvis-architect .`). Все 8 DNA-файлов уже лежат в корне проекта **с плейсхолдерами в фигурных скобках** — например `{{NAME}}`, `{{ROLE}}`. Твоя задача — провести интервью из 10 вопросов и заменить эти плейсхолдеры на реальные ответы пользователя.

## Как ты задаёшь вопросы

Если в твоей среде есть инструмент структурированного вопроса (например `AskUserQuestion` или аналогичный picker с вариантами) — **используй его**. Передавай предложенные варианты как опции. Это даст пользователю красивый виджет с кнопками.

Если такого инструмента нет (обычный текстовый чат Claude Code) — **выводи варианты в чат нумерованным списком** в самом тексте вопроса. Пользователь ответит номером или своими словами.

В обоих случаях — **по одному вопросу за раз**, не списком из десяти.

## Что нужно сделать (по шагам)

### Шаг 1. Поздоровайся и объясни что будешь делать (1 минута)

Скажи примерно так (адаптируй под пользователя, не копируй дословно):

> «Привет. Я скачал шаблон Вашего Агента — все файлы уже в проекте. Сейчас задам 10 коротких вопросов про Вас и Ваши цели, чтобы заполнить шаблоны под Вас. Это 5–7 минут. Готовы?»

Дождись подтверждения. Если ученик не готов сейчас — скажи что инструкция остаётся в файле `INSTALL.md`, можно вернуться позже.

### Шаг 2. Прочитай 8 DNA-файлов в корне

Файлы (в корне проекта, не в подпапке):
- `CLAUDE.md` — правила работы Агента
- `SOUL.md` — личность Агента
- `MEMORY.md` — блокнот фактов
- `GOALS.md` — цели
- `USER.md` — профиль владельца
- `MISSION.md` — главная миссия
- `PROJECTS.md` — активные проекты
- `PREFERENCES.md` — предпочтения и ограничения

В каждом найдёшь плейсхолдеры в двойных фигурных скобках: `{{NAME}}`, `{{ROLE}}` и т.д. Их нужно заменить ответами пользователя на интервью.

В корне проекта в каждом из восьми файлов **замени все `{{плейсхолдеры}}` на реальные ответы**.

**Маппинг плейсхолдеров по файлам:**

| Плейсхолдер | Файлы где встречается |
|---|---|
| `{{NAME}}` | CLAUDE.md, SOUL.md, MEMORY.md, USER.md |
| `{{ROLE}}` | SOUL.md, USER.md |
| `{{ROLE_DESCRIPTION}}` | MEMORY.md, USER.md |
| `{{ACTIVE_PROJECTS}}` | MEMORY.md, PROJECTS.md |
| `{{Q_GOAL}}` | GOALS.md, MISSION.md |
| `{{ADDRESS_FORM}}` | CLAUDE.md, SOUL.md, USER.md, PREFERENCES.md |

Файлы остаются на своих местах в корне — они никуда не двигаются. Просто заполняются.

**Особое внимание к `MEMORY.md` — он структурирован по 8 разделам:**

| Раздел в MEMORY.md | Какие плейсхолдеры туда |
|---|---|
| Кто я | `{{NAME}}`, `{{ROLE}}`, `{{ROLE_DESCRIPTION}}` |
| Контакты и аккаунты | оставь пустым (пользователь дополнит руками) |
| Активные проекты | `{{ACTIVE_PROJECTS}}` |
| Инфраструктура | оставь заглушку *«Пока пусто. Появится когда заведёте свой сервер или домен»* |
| Стек и инструменты | оставь заглушку *«Пока пусто. Появится по мере появления конкретного стека»* |
| Предпочтения | `{{ADDRESS_FORM}}`, `{{ANSWER_STYLE}}`, `{{LANGUAGE}}`, `{{IRRITATIONS}}` |
| Ключевые решения | оставь заглушку *«Пока пусто. Появится по мере работы»* |
| Ссылки на knowledge/ | оставь заглушку *«Пока пусто»* |

### ⚠️ Важно про русские имена и падежи

Шаблоны переписаны так, чтобы плейсхолдер `{{NAME}}` встречался только в **именительном падеже** (например *«Зови меня Дмитрий»*, *«Имя владельца: Дмитрий»*). Простой замены `{{NAME}}` → ответ пользователя достаточно.

Но если **где-то найдёшь конструкцию которая требует склонения** (например в косвенных падежах — «работаю с …», «не помнит …»), и простая замена даст неестественный текст — **подставь имя в правильной форме**. Не оставляй грамматически кривой русский.

Примеры:
- «спрашивай Дмитрия напрямую» (винительный) — НЕ «спрашивай Дмитрий напрямую»
- «я работаю с Дмитрием» (творительный) — НЕ «я работаю с Дмитрий»
- «дневник Дмитрия» (родительный) — НЕ «дневник Дмитрий»

Сейчас таких мест в шаблонах нет, но при будущих расширениях могут появиться.

### Шаг 5. Удали HTML-комментарий «WIZARD»

В каждом из 8 DNA-файлов в самом начале есть комментарий-маркер:

```

```

После того как все плейсхолдеры заменены — **удали этот комментарий из всех 8 файлов**. Это сигнал «установка завершена».

Прежде чем удалять служебные папки и делать коммит — **покажи пользователю в чате превью каждого заполненного файла**:

> «Вот что получилось — посмотрите. Если где-то поправить — скажите, я перепишу.»

Покажи поочерёдно содержимое (можно группами по 2-3):

2. `USER.md` — профиль
3. `MISSION.md` — миссия
4. `CLAUDE.md` — правила работы
5. `MEMORY.md` — что уже знает Агент
6. `GOALS.md` — цели

Дождись подтверждения «всё ок» или правок. Если правки — внеси их и снова покажи.

### Шаг 7. Удали служебные файлы

После подтверждения пользователя выполни **одну команду**:

```bash
git rm -r INSTALL.md README.md examples/
```

> **Почему `git rm` а не `rm -rf`:**
> Если пользователь работает в режиме «Ask Before Edit» — команда `rm -rf examples/` будет заблокирована Claude Code как потенциально опасная (RED-операция). Каждый файл придётся подтверждать отдельно.
>
> `git rm` — это git-команда (управление staging area + физическое удаление), не shell-удаление. Она проходит без блокировки, удаляет всё одним шагом и сразу подготавливает удаление к коммиту.

Объясни простыми словами: *«Удалил служебные файлы — они были нужны только для установки. В проекте остались 4 главных файла Агента, папки `memory/` и `knowledge/`, папка `.claude/` с настройками и скиллами, и `.gitignore`».*

> **Важно:**
> - Папки `memory/` и `knowledge/` **остаются** — это рабочие папки для дневников и справочников. Их `README.md` файлы тоже остаются — они объясняют пользователю как с ними работать.
> - Папка `.claude/` **остаётся** — там settings.json (настройки разрешений) и skills/ (4 предустановленных скилла).
> - Папка `examples/` удаляется целиком — она была только для подсматривания.

**Если `git rm` всё-таки не сработал** (например проект ещё не под git, что для нашего курса нетипично — у пользователя git настроен с Урока 3):

```bash
rm INSTALL.md
rm README.md
find examples/ -type f -delete
find examples/ -type d -empty -delete
```

`find ... -delete` — обходной путь который не блокируется как `rm -rf`.

### Шаг 8. Первый коммит и push

Объясни что делаешь. Один коммит, не дроби:

```bash
git add -A
git commit -m "Архитектура Агента собрана"
git push
```

Если у пользователя ещё нет git remote (Урок 3 он мог пройти без push) — спроси и помоги настроить.

## Чего НЕ делать

- **Не показывай заполненные файлы пользователю до того как все 10 вопросов отвечены.** Сначала всё интервью, потом превью.
- **Не делай больше 10 вопросов** — это интервью, не допрос.
- **Не заполняй плейсхолдеры догадками.** Если ответ непонятен — переспроси одной фразой.
- **Не пропускай Шаг 6** (показ превью). Это критическая точка контроля пользователя.
- **Не удаляй HTML-комментарии WIZARD до Шага 5** — они защищают от ситуации «открыли Claude Code в недоустановленном проекте».
- **Не удаляй `.git/`** или `.gitignore` — они нужны.
- **Не сохраняй ответы интервью в отдельный файл** (типа `interview-answers.md`). Они должны попасть только в шаблоны.
- **Не запускай повторный `npx degit`** — он перезапишет уже заполненные файлы.

## Если что-то пошло не так

- **Пользователь хочет начать заново** → скажи что лучше всего удалить папку и сделать `npx degit Ntmib/jarvis-architect .` снова. Не пытайся «откатить» вручную.
- **Пользователь не знает что отвечать на открытый вопрос** → дай 2-3 варианта на выбор. Например для «роли»: «предприниматель / специалист / ученик / создатель контента — что ближе?»
- **Пользователь хочет посмотреть пример заполненного файла** → покажи `examples/jarvis-example/SOUL.md`. Там реалистичный пример (предприниматель Алексей).
- **Если ты Claude Code в режиме Ask Before Edit** — для команд `rm`, `git` нужно подтверждение. Объясняй пользователю что собираешься сделать перед каждой такой командой.
- **Если установка прервалась на середине** (закрыли терминал, потеряли сессию) → пользователь снова открывает Claude Code в этой папке, и ты видишь оставшиеся `{{плейсхолдеры}}` в файлах. По HTML-комментариям WIZARD понятно что установка не закончена. Скажи «Вижу что установка прервалась — продолжим с того места где остановились?» и догони оставшиеся вопросы.
